# Контрибьютинг в EndfieldPass

Спасибо за вклад в проект.

Официальный сайт: **https://endfieldpass.site/**

Языковые версии:

- Русская: [CONTRIBUTING](CONTRIBUTING.md)
- English: [CONTRIBUTING_EN](CONTRIBUTING_EN.md)

## Перед началом

- Проверь, нет ли уже похожего `Issue` или `PR`.
- Для крупных изменений сначала открой `Issue`, чтобы согласовать ожидания.
- Формулируй изменения конкретно: что, зачем и как проверить.

## Как оформлять Issue

Для баг-репорта укажи:

- шаги воспроизведения
- ожидаемое поведение
- фактическое поведение
- скриншоты/видео (если это UI)
- окружение: ОС, браузер, версия Python, ветка/коммит

Для feature-request укажи:

- проблему пользователя
- предлагаемое решение
- компромиссы и альтернативы

## Как оформлять Pull Request

1. Создай отдельную ветку от `main`.
2. Делай небольшие и логически связанные коммиты.
3. Добавляй или обновляй тесты, если меняется поведение.
4. Прогони локальную проверку перед открытием PR.
5. В описании PR укажи что изменено, как протестировано и на что смотреть ревьюеру.

## Локальная проверка

```bash
python -m venv .venv
# Windows:
.venv\Scripts\activate
# Linux/macOS:
source .venv/bin/activate

pip install -r requirements.txt
python manage.py migrate
python manage.py test
```

## Рекомендации по изменениям

- Не смешивай в одном PR несвязанные рефакторинги и новую функциональность.
- Если меняется UI, прикладывай скриншоты `before/after`.
- Если меняется текст интерфейса, проверь ключи локализации.
- Никогда не коммить секреты (`.env`, токены, OAuth-ключи, персональные данные).

## Чеклист PR

- [ ] Проект локально запускается без регрессий
- [ ] Тесты проходят
- [ ] Документация обновлена (если нужно)
- [ ] В diff нет секретов и лишних артефактов
- [ ] Описание PR понятно и достаточно для быстрого ревью
